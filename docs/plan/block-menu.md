# 従量ブロック メニュー（マスタ）

> 2026-07-09 新設。`service-operations.md §7` から明細を分離したマスタデータ。
> **設計思想（なぜ従量を出口に足すか・覆さない原則5つ・階段の位置づけ）の正は `service-operations.md §7`。** ここは「メニュー本体（明細）」の正。ブロックは今後増やし続ける想定なので、増える明細と増えない思想を分けている。
> 前提: `service-operations.md`（背骨は不変・従量は既存客の出口限定）／`pricing.md`（単価確定はこちらと連動）。

## 列の意味

| 列 | 中身 |
|---|---|
| **ID** | 系統別の連番（IMP=実装／MNG=管理・資産／SRV=調査）。付けたら変えない。 |
| **ブロック** | 課金単位（中粒度）＝「1つの完結した状態変化」。 |
| **含む小作業（顧客提示の内訳）** | 顧客に透明性として見せる内訳。**小は個別課金しない**（便利屋化を防ぐ）。 |
| **完結成果（客の一言）** | 売るときの価値の言い方（ベネフィット）。 |
| **完了条件（客観判定）** | 検収・請求のトリガー。**「状態」でなく客観条件で定義する**（解釈ブレ・検収もめを防ぐ）。 |
| **必要スキル／対応区分** | 誰に振れるか。★＝本人にしかできない／◆＝外注可能（スポット）／▲＝資産化・AIで圧縮。DIR＝本人のディレクション工数。 |
| **単価（仮）** | すべて仮置き（本人が確定）。◆実装は「外注実費＋DIR」で透明化（マージンを取らない）。 |

## ブロックを増やすときのルール

1. **系統を先に決める**（①実装／②管理・資産／③調査のどれか）。3系統に入らない＝背骨（★）の可能性。従量メニュー化しない。
2. **IDを採番**（系統プレフィックス＋連番。既存IDは変えない）。
3. **7列すべて埋める**。特に **完了条件は必ず客観判定**で書く（状態語で終えない）。
4. **単価は仮のまま置いてよい**が、確定は `pricing.md` と連動。◆実装は「外注実費＋DIR」を崩さない。
5. **依存の歯止め**（`service-operations.md §7-4`）: 実装系の反復依頼が続く業務は、管理・資産ブロック（引き継ぎ可能化＝MNG-01系）へ設計変更を促す。

## ▶要確定（本人）

- 各ブロックの単価の当たり（外注実費＋DIRの相場感）。
- **対応先候補**（人/企業）の列を将来足すか——今は「必要スキル」まで。外注ネットワークが具体化したら `対応先候補` 列を追加。
- プル型で実際に来た追加依頼はどのIDに当たるか／不足ブロックはどれか。
