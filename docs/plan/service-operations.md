# サービス実務の分解 — 本人がやる業務／外注する想定業務（たたき台）

> 作成: 2026-07-03 / ステータス: DRAFT v1（本人が違和感を起点に削る前提）
> 目的: ①サービスの具体（いつ・何をやるか）を実務レベルまで落とす ②各タスクを「本人コア／外注可／資産化」に分類する。これが `pricing.md`（料金の原価根拠）と、外注導入の判断材料（コンサル指針 WAIT の解除条件）になる。
> 前提: `service-definition.md`（商品の中身・契約）／`target-conditions.md`（顧客条件）／HANDOFF 13章（人員=WAIT。「本人にしかできない判断」と「外部に渡せる作業」を分解し、スポット補助から）。
> 凡例: **★＝本人にしかできない**（構造の見立て・判断・経営者との関係）／**◆＝外注可能（スポット）**／**▲＝資産化・AIで圧縮**（テンプレ・自動化＝L-3。同価格で稼働半減の源泉）

## 4. 逓減期・ライト

- **逓減期**: 定例を隔週に半減。課題ボードは相手が回し、本人は見守り＋要所の介入のみ。→ 竹の約半分の工数（月20〜25h）
- **ライト**: 月1定例（1h）＋チャット＋四半期レビュー（半日）。→ 月5〜8h。**システム保守・実作業はライト月額に含まない**（含めると代行に戻る）。手を動かす実作業が要るときは**従量ブロック（§7・`block-menu.md`）で別建て**。

## 6. 外注の設計原則（WAIT との整合・覆さない）

1. **固定費化しない**: 月額契約（伴走）1社が現金化するまで外注ゼロ。以降も件数・時間単位のスポットのみ。正社員・固定外注・営業代行はしない（コンサル指針＋HANDOFF 13章）。
2. **商品の成立に外注を組み込まない**（HANDOFF 7章）: 外注が全員消えても診断・伴走は回る。外注は「本人の時間を買い戻す」ためだけに使う。
3. **マージンを取らない**（method.md・非依存の担保）: 実装外注は実費ベースで顧客に開示。ただし**ディレクション（要件定義・検収）は自分の実工数の対価として別建てで取る**——中抜きはしないが、タダ働きもしない。
4. **渡す順番**（時間単価の低い順・失敗コストの低い順）:
   - ①文字起こし・清書・一覧化（オンラインアシスタント） — 想定 2,500〜4,000円/h・月5〜10h＝月2〜4万
   - ②図版・資料整形（クラウドソーシング／織物屋系デザイナー） — 1〜3万/件
   - ③実装（コーダー・副業の隙間稼働＝既存ネットワーク） — 想定 5,000〜8,000円/h・案件単位
   - ④請求・調整などの事務（③まで回ってから）

**媚びない注記**: §2〜3の工数を見ると、**外注で買い戻せるのは月10h前後**が当面の上限（★が大半のサービス設計だから。それ自体は非依存・属人価値の裏返しで正しい）。「外注で伴走4社目」は成立しない＝**3社上限は外注では動かない**。動かすなら「▲資産化」で自分の工数を圧縮する方が先。

## 7. 従量ブロックメニュー（出口の追加・3系統｜設計思想）

> 2026-07-08 追加（たたき台）。**プル型で「手を動かして支援してほしい」実需が出た**ことを受け、月額固定一択（HANDOFF 4章）を一部見直し。ただし **背骨は不変**——従量は「診断・濃い伴走を通った既存客が、ライト土台の上に積む追加」に限定し、**従量単独・従量入口は作らない**。
> **ブロックの明細（メニュー本体）は `block-menu.md`（マスタ）に分離**（2026-07-09。今後増やし続ける＋対応スキル・単価を持たせるため）。ここには増えない「設計思想」だけを残す。

### 7-0. 覆さない設計原則（5つ）

1. **★背骨（構造の見立て・判断・経営者との関係）はブロック化しない。** 従量にしてよいのは ◆実装／▲管理・資産／調査 の運用実働のみ（背骨を従量メニュー化すると便利屋化して事業が溶ける）。
2. **課金は成果／ブロック単位（時間で売らない）。** 速くやって損しない。上限を作らないので追加依頼しやすい（＝クライアントの「これもお願い」に応えられる・Win-Win）。
3. **軽微な相談・微修正はブロックにせず、ライト月額に含む**（雑務の時間労働化を防ぐ）。
4. **◆実装を客に出すときは「外注実費＋ディレクション工数」で透明化**（マージンを取らない・method.md）。
5. **調査ブロック ≠ Phase0診断。** 診断＝全体構造の見立て（背骨）／調査ブロック＝方向が決まった中での個別領域の実働。

### 7-1. 商品階段の中での位置づけ

```
無料相談 → 診断 → 濃い伴走（構造を立て直す＝必須の背骨）→ 定着 → ライト土台（月額・見守り＋軽微な相談/微修正）
                                                                        │
                                                                        └─▶ 従量ブロックを都度積み増し（既存客のみ・上限なし）
                                                                             ├ ① 実装
                                                                             ├ ② 管理・資産
                                                                             └ ③ 調査
```

### 7-2. 提示ルール（課金＝中粒度／説明＝小の内訳）

- **課金単位はブロック（中）**＝「1つの完結した状態変化」。**顧客には中に含む小作業を内訳として見せる**（透明性）。小を個別課金しない（＝便利屋化を防ぐ）。
- **単価はすべて仮置き（本人が確定）**。◆実装は「外注実費＋ディレクション（DIR）」の建て付け。
- **完結は「状態」でなく「客観条件」で定義する**（例：導入セットアップ＝“使える状態”ではなく“設定条件表の全項目が設定済”）。状態語は解釈がブレて検収でもめる。各ブロックの完了条件は `block-menu.md` の「完了条件（客観判定）」列で管理。

### 7-3. ブロック明細 → `block-menu.md`（マスタ）

- 3系統（①実装／②管理・資産／③調査）の各ブロック・内訳・**完了条件（客観判定）**・必要スキル・単価は **`block-menu.md` を正**とする。
- ブロックの追加・単価確定もそちらで行う（今後増やし続ける想定のため明細を分離）。

### 7-4. 依存させないための歯止め

- **管理・資産／調査系はやるほど客に資産が残る＝自走に近づく（非依存）。** 積み増し自由でよい。
- 危ないのは **実装系を手を動かし続ける形**。§3の判断——**1回きり＝外注／繰り返す＝資産化して管理系へ転換**——で歯止め。実装ブロックの反復依頼が続く業務は、管理・資産ブロック（引き継ぎ可能化）へ設計変更を促す。
